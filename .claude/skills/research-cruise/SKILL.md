# research-cruise

사용자가 제공한 크루즈 일정을 파싱하고, 서브에이전트로 선박·관광지 정보를 인터넷에서 조사하여 workspace에 저장한다. 파일 생성은 하지 않는다 — 파일 생성은 `/generate-cruise`에서 수행한다.

---

## 작업 공간 (Workspace)

### 디렉토리 구조

`.cruise-workspace/{slug}/`를 프로젝트 루트에 생성한다:

```
.cruise-workspace/{slug}/
├── parsed-data.md          ← 1단계 파싱 결과
├── agent-s.md              ← Agent S 조사 결과
└── agent-p-{도시명}.md     ← Agent P 조사 결과 (도시별)
```

### 재개 감지

각 단계 시작 시 기존 작업 상태를 확인하여 완료된 작업을 건너뛴다. 인터넷 조사는 비용이 크므로, 이미 완료된 결과를 재활용하는 것이 효율적이다 (generate-cruise는 로컬 파일 생성이라 전부 재생성하지만, research-cruise는 의도적으로 건너뛰기를 유지한다):

| 확인 대상 | 존재하면 |
|-----------|----------|
| `.cruise-workspace/{slug}/parsed-data.md` | 1단계 건너뜀 — 파일에서 파싱 데이터 읽기 |
| `.cruise-workspace/{slug}/agent-s.md` | Agent S 건너뜀 |
| `.cruise-workspace/{slug}/agent-p-{도시명}.md` | 해당 Agent P 건너뜀 |

## 1단계: 사용자 데이터 파싱

### slug 결정

사용자 데이터에서 크루즈 지역명을 추출하여 slug를 결정한다: `{지역명 영문 소문자}-cruise`
- 예: 하와이 → `hawaii-cruise`, 아라비아 → `arabia-cruise`, 북유럽 → `north-europe-cruise`
- slug는 workspace 경로(`.cruise-workspace/{slug}/`)와 이후 generate-cruise에서의 데이터 디렉토리 경로(`data/products/{slug}/`)에 사용된다.

### 데이터 추출

사용자가 제공한 일정(텍스트 또는 이미지)에서 다음 정보를 추출:
- 선박명 및 크루즈 선사
- 출발일 및 기간 (N박 N일)
- 항공사 및 항공편 정보 (직항 vs 경유, 제공된 경우)
- 기항지 도시 및 해상일
- **각 일차별 개별 관광지/체험 이름 목록** (이름이 붙은 관광명소, 박물관, 사원, 전망대, 체험 투어 등을 모두 추출 — 이 목록이 tourist-spot 카드와 모달의 생성 근거가 됨)
- 가격 데이터 (제공된 경우)
- 기항지 선택 관광 정보 (제공된 경우)

**파싱 결과 저장:** 추출한 데이터를 `.cruise-workspace/{slug}/parsed-data.md`에 구조화하여 저장한다. 포함 항목:
- 기본 정보 (slug, 선박명, 선사, 출발일, 기간, 항공사)
- 항공편 (편명, 출발/도착 시간, 소요시간)
- **일차별 상세 일정** (날짜, 위치, 시간별 이벤트, ▣ 세부항목 전체, 관광지명, 식사, 숙박) — schedule-days 생성의 핵심 데이터이므로 사용자 원문의 충실도를 유지
- 기항지별 관광지 목록 (Agent P 프롬프트용)
- 가격 데이터 (제공된 경우)

## 2단계: 서브에이전트 인터넷 조사 (병렬 실행)

**재개 확인:** `.cruise-workspace/{slug}/`에 에이전트 결과 파일(`agent-s.md`, `agent-p-*.md`)이 이미 존재하면, 해당 에이전트를 실행하지 않고 파일에서 결과를 읽는다.

**병렬 에이전트 아키텍처**를 사용한다. 선박 조사 1개 + 기항지별 조사 N개를 전부 병렬 실행한다:

```
[병렬 실행]
├── Agent S: 선박 조사 (제원 + 시설 + 객실) — 공식 사이트 기반
├── Agent P-1: 기항지 1 조사 (예: 두바이 — 도시 개요 + 관광지 3개)
├── Agent P-2: 기항지 2 조사 (예: 무스카트 — 도시 개요 + 관광지 2개)
├── Agent P-3: 기항지 3 조사 (예: 도하 — 도시 개요 + 관광지 2개)
└── Agent P-4: 기항지 4 조사 (예: 아부다비 — 도시 개요 + 관광지 3개)
```

각 기항지 에이전트가 해당 도시의 개요 + 모든 개별 관광지를 함께 조사한다. 같은 도시의 관광지는 교통, 문화, 지리 등 공유 컨텍스트가 있어 하나의 에이전트가 처리하는 것이 품질과 효율 면에서 최적이다.

**해상일(sea day)**은 별도 에이전트를 배정하지 않는다. Agent S의 선박 시설 조사 결과를 활용하여 생성한다.

**결과 저장:** 각 에이전트 프롬프트에 workspace 파일 저장 지시를 포함한다. 프롬프트 구성 시 `{workspace}`를 프로젝트 루트 기준 절대 경로(`{프로젝트_루트}/.cruise-workspace/{slug}`)로 치환한다. 에이전트 결과를 받으면 해당 파일이 정상 저장되었는지 확인하고, 누락 시 메인 에이전트가 직접 저장한다.

> **별도 검증 에이전트 없음**: Agent S 프롬프트 자체에 공식 사이트 필수 + 확인 불가 시 빈 문자열 규칙이 포함되어 있으므로, 별도의 검증 단계 없이 Agent S 결과를 그대로 사용한다.

**Agent S — 선박 조사 (공식 사이트 필수):**
크루즈 선박의 정보를 **해당 선사의 공식 사이트**에서 우선적으로 수집한다:
- 선박 제원: 톤수, 길이, 승무원, 높이(미터), 갑판 수(층), 승객 정원, 첫 항해, 객실 수
- 시설 설명: 엔터테인먼트, 아웃도어 & 레저, 힐링 & 웰니스 (3개 카테고리)
- 객실 정보: **공식 사이트에 기재된 객실 타입별 어메니티를 있는 그대로** 수집 (임의 추가/생략 금지)

```

- 모든 데이터 항목에 출처 URL을 기록할 것
- 공식 사이트에 명시되지 않은 시설·객실 항목은 포함하지 말 것 (제원은 아래 폴백 규칙 참조)

   - 힐링 & 웰니스: 스파, 피트니스 센터, 사우나, 자쿠지

   ※ 한국 일반 소비자가 바로 이해할 수 있는 단어만 사용 — 생소한 외래어(리트릿, 트리트먼트 등)는 쉬운 한국어로 대체

   a) 공식 사이트에서 사용하는 정확한 객실 타입 명칭

   c) 공식 사이트에 기재된 어메니티/설비를 있는 그대로 전부 수집
```

## 완료

모든 에이전트 조사가 완료되면 다음을 확인하고 안내한다:

1. `.cruise-workspace/{slug}/` 내 파일 목록을 출력하여 저장 상태 확인
2. 사용자에게 안내:

```
조사가 완료되었습니다.
저장된 파일: parsed-data.md, agent-s.md, agent-p-{도시1}.md, ...

다음 단계: `/clear` 후 `/generate-cruise`를 실행하세요.
```
